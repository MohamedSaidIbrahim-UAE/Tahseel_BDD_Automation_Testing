# Test Plan – Reports Module  
**Product:** Tahseel Manage Portal (Staging)  
**Version:** As per staging environment (2026-06-03)  
**Focus Area:** All report categories, with priority on Financial and External Systems reports  
**Plan Owner:** QA Lead  
**Date:** 2026-06-05  

---

## 1. Introduction  
### 1.1 Purpose  
This test plan defines the strategy, scope, approach, and focus areas for testing the Reports section of the Tahseel Manage Portal. The module contains over 100 reports covering transactions, payments, refunds, deposits, loading, revenue, tax, settlements, and operational logs. Given the financial nature of the data, testing must ensure absolute accuracy, security, performance, and compliance with business rules.

### 1.2 Objectives  
- Validate that all reports render correct, complete, and consistent data.  
- Ensure role-based access controls (RBAC) enforce strict report visibility.  
- Confirm business rules (fees, revenue sharing, tax, adjustments) are correctly applied in report calculations.  
- Verify filter logic, drill-down, export, and print formats.  
- Test integration with internal databases and external bank/entity APIs for settlement reports.  
- Identify and handle edge cases: broken approval chains, complex multi-entity transactions, missing data, large datasets.  
- Cover functional, performance, security, and localisation testing for the Reports module.

## 3. Test Approach & Strategy  

### 3.1 Overall Methodology  
- **Risk-based testing:** All P1 reports undergo rigorous data reconciliation; P2/P3 reports receive sanity and boundary testing.  
- **Incremental validation:** Start with smoke tests (report opens, filters work), then in-depth data accuracy, then edge cases.  
- **Data-driven testing:** Seed a controlled dataset (e.g., 50 transactions with known values, fees, tax) to validate calculations precisely.  
- **End-to-end integration:** For external settlement reports, compare generated report files with simulated bank response files.

### 3.2 Test Design Techniques  
- Equivalence partitioning and boundary value analysis for date/numeric filters.  
- Decision table testing for fee/tax calculation rules (e.g., service type, entity, payment method).  
- State transition for approval status reflected in reports (e.g., refund request pending → approved → completed).  
- Orthogonal arrays for multi-filter combinations.

## 4. Focus Areas (Per Request)  

### 4.1 Role-Based Access Control (RBAC)  
**Objective:** Ensure each user role sees only the reports assigned in the menu JSON and cannot access others via URL manipulation.

**Test Strategy:**
- Prepare a matrix: roles (Gov User, Smart User, Operator, Company, Money Loader, Merchant, Seller, etc.) vs. all report groups.
- For each role, login and verify the sidebar menu shows exactly the expected reports (use the provided `prevId` logic and submenu membership).
- Attempt direct navigation to a report URL (e.g., `/report-show/...`) that is not authorised – expect 403 or redirect to dashboard with “Access Denied”.
- Verify that report parameters (entity filter, scope) are automatically restricted based on user’s scope/role (e.g., an entity-specific user should only see that entity’s data even if filter dropdown suggests otherwise).

**Negative scenarios:**  
- Unauthorised role tries to export/print – action blocked.  
- User with expired session – redirect to login, no report data cached.

### 4.2 Related Rules Applications  
**Objective:** Confirm business rules governing fees, revenue sharing, VAT, and special conditions are correctly reflected in report totals and breakdowns.

**Rules tested include:**
- Service fees per entity/service type.  
- Revenue sharing percentages between entities (e.g., Sharjah Municipality and Town Planning).  
- VAT (5%) calculation: inclusive/exclusive treatment, tax invoice accuracy.  
- Direct debit fee logic.  
- Load/deduct service charges.  
- Deposit retain/confiscation rules.

**Approach:**
- Identify the rules database/service rules configuration page (already in menu: “ServiceRules”).  
- Create transactions that trigger specific rules; run the corresponding report (e.g., Revenue by Entity) and verify amounts match rule-engine outputs.  
- Modify rules on staging, verify reports reflect updated calculations.  
- Test edge cases: zero-fee services, tax-exempt entities, transactions with multiple fee components.

### 4.3 Calculations & Prints  
**Calculations:**
- For every financial report, re-calculate sums, subtotals, grand totals, and fee breakdowns using raw database queries (or API responses) as the source of truth.  
- Validate aggregation: daily, monthly, yearly, by entity, by payment method.  
- Test rounding and decimal precision (e.g., AED amounts to 2 decimal places).  
- Check computed fields: net amount, VAT amount, total including VAT.

**Prints / Exports:**
- Test PDF and Excel export for sample reports; verify layout matches on-screen table.  
- Ensure Arabic and English prints correctly (RTL for Arabic, appropriate fonts, date formats).  
- Test print with large datasets (pagination, headers/footers repeated).  
- Negative: export with no data returns a clear empty message, not a crash.

### 4.4 Filters  
**All filter types to test:**
- Date range (from/to), quick picks (today, this month, custom).  
- Entity / department / revenue entity dropdowns.  
- Payment method (credit card, cheque, e-voucher, etc.).  
- Transaction status (success, cancelled, pending).  
- Account/ card/ cheque number.  
- User or collector ID.  

**Positive tests:**  
- Combine multiple filters; verify result set matches SQL query with same WHERE clause.  
- Switch between filters without page refresh, reset filters to default.  
**Negative tests:**  
- From date > To date – clear error.  
- Non-existent entity – “No data found”.  
- Special characters in text filters (SQL injection attempt).  
- Very large date range that spans multiple partitions – check performance and completeness.

### 4.5 Professional Test Scenarios (Positive & Negative)  
This section provides high-level scenarios that will be detailed in test cases.

#### 4.5.1 Broken Approval Chains  
Context: Reports like “Refund Detailed Report” or “Adjustment Register Requests” reflect workflow statuses.  
- **Scenario:** A refund request is stuck in “Approval Required” because an approver left the organisation. The report should still display the transaction with its correct pending status, not omit it or show inconsistent state.  
- **Scenario:** An adjustment request is rejected after partial processing. The adjustment register report must show the latest status and no financial impact until re-approved.  
- Test that reports based on `workflowManag` tables handle NULL assignee, missing users, and forcibly terminated workflows.

#### 4.5.2 Complex Behaviours  
- **Multi-currency-like scenarios:** Though all in AED, test handling of different decimal scales (3-digit decimals for some fees).  
- **Concurrent report generation:** Fire multiple heavy reports simultaneously; ensure each completes without data mixing.  
- **Asynchronous reports:** If a report is generated asynchronously (e.g., large settlement file), test the queue, progress indicator, and download availability.  
- **Time zones & daylight saving:** Reports must respect server timezone (Gulf Standard Time) even when generated from a browser in a different timezone.

#### 4.5.3 Adjustments Master Data and Transactions  
- **Master data:** Create a new adjustment type via backend (if possible) and verify the “Adjustment Register Requests” report picks up the new type in its filters/legend.  
- **Transactions:** Post an adjustment that increases a Tahseel account balance; then run the “Tahseel Card Transaction Details Report” – the adjustment must appear as a distinct entry with the correct amount and reference.  
- Run “Aggregated fees Report Tahseel Cards” and ensure adjustment fees (if any) are counted.  
- Negative: Delete an adjustment from the database directly (simulate data integrity issue) – the report must either display an error or exclude the orphan row gracefully, not crash.

### 4.6 Integration with Database and API Data (Internal & External)  
**Internal Integration:**
- Compare report data directly with staging database via SQL queries.  
- For each report, identify the source tables/views; trace the data lineage.  
- Test with a snapshot of production-like data (anonymised) to catch discrepancies in aggregations.  
- Validate that report pages call internal APIs with correct parameters; intercept and replay with modified parameters (API testing).

**External Integration (Settlement Reports):**
- **Mashreq Bank Settlement:**  
  - Simulate bank response file (CSV/XML) containing a set of transactions.  
  - Run “Mashreq Settle Report” and “Mashreq bank settlement report in period” and verify totals match the file.  
  - Test mismatch scenarios: bank file has extra transactions – report must show discrepancies; missing transactions – report must highlight.  
- **Sharjah Islamic Bank (SIB) Settlement:** Repeat as above with SIB format.  
- **UAE Central Bank Settlement:** Similar, ensure compliance with Central Bank file standards.  
- **Negative:** Corrupt external file (truncated, wrong encoding) – system should log error and notify, not display corrupted data.

## 5. Test Types  

### 5.1 Functional Testing  
- **Smoke testing:** Quick run of all P1 reports after deployment.  
- **Detailed functional:** All scenarios described in Section 4.  
- **Regression:** Automated scripts for P1 report calculations after any code change.

### 5.2 Performance Testing  
- **Load test:** Simulate 100 concurrent users running different reports (including large settlement reports). Validate response time <10 seconds for standard reports, <30 seconds for heavy aggregations.  
- **Stress test:** 500 concurrent report requests – system must queue or gracefully degrade, no crashes.  
- **Volume test:** Execute reports over a dataset with 1 million transactions; check memory usage and completion.  
- **Endurance test:** Continuous report generation for 2 hours; monitor resource leaks.

### 5.3 Security Testing  
- **Injection attacks:** Attempt SQL injection in all filter fields; XSS in report name/description (if editable).  
- **Authorisation bypass:** Direct API call to report endpoint with another user’s session token – reject.  
- **Sensitive data:** Ensure no credit card full numbers, bank account details appear in reports unless properly masked.  
- **Audit trail:** Each report generation should be logged (user, timestamp, parameters) – verify using system logs.

### 5.4 Localisation Testing  
- Switch language to Arabic; verify all report titles, column headers, filter labels, and data formatting (dates in Hijri? Gregorian? – confirm locale).  
- Check RTL layout for Arabic – menus, filter panels, table columns should flip appropriately.  
- Verify number formats: Arabic numerals vs. Hindi numerals, currency symbol placement.  
- Export PDF/Excel in Arabic – ensure content is readable and not reversed.
