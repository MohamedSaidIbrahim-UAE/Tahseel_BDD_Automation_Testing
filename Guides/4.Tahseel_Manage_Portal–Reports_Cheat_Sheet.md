# Tahseel Manage Portal – Reports Cheat Sheet  
**Professional Testing Reference**  
*Grouping by shared business objectives, testing focus, and common test conditions*

---

## 1. Reconciliation & Settlement Reports  
**Shared Objective:** Match internal transaction records with external bank/payment gateway files, identify discrepancies, and produce settlement summaries.

**Included Reports (all Financial group):**
- Mashreq Settle Report / in Period
- Bank Transfers of Sharjah Islamic Bank / General / in Period
- UAE Central Bank Settlement Report / in Period
- Sharjah Islamic Bank Settlement Report / in Period
- Intensive Settlement Details Report
- Fedral Bank Transfer Detail Report
- Bank Transfers of Transaction Fees – Receipts

### Common Test Scenarios & Conditions
| **Positive Scenarios** | **Negative / Edge Scenarios** | **Key Conditions to Verify** |
|------------------------|-------------------------------|------------------------------|
| Full match: all internal transactions found in bank file with identical amounts, dates, references | Missing record: internal transaction not in bank file → flagged “Unmatched” | Reconciliation difference must be 0 for full match |
| Partial match: transactions with same reference but amount mismatch → highlighted with variance | Extra record in bank file → flagged “Bank Only” | Amount precision: variances of 0.01 AED must be caught |
| Settlement in period: aggregations per day, grand totals correct | Corrupt bank file (wrong encoding, missing columns) → system rejects with clear error | File format validation; mandatory column presence |
| Large volume (50k+ records) performance: report generates within SLA | Duplicate processing – system prevents double counting | Masking of sensitive data (account numbers) in report output |
| Export to PDF/Excel preserves all columns and totals | Future date → “No data found” | Audit log of report generation recorded |

## 2. Financial Calculation & Tax Reports  
**Shared Objective:** Compute and present VAT, service fees, transaction fees across payment methods, and entity-specific fee shares according to business rules.

**Included Reports:**
- All **Tax Reports** (Transaction Details/Summary Tax, by Revenue Entity, Tax Invoice)
- Transaction Fees For All Payment Methods
- Financial Services Report
- Grant Card History Logs Report (fee aspects)
- Aggregated/Detailed Reports of Smart Receipt Fees
- E‑Voucher Report Transaction
- Municipality Fees Share Report on Real Estate Registration
- Total Service Charges for Loading Transactions

### Common Test Scenarios & Conditions
| **Positive Scenarios** | **Negative / Edge Scenarios** | **Key Conditions to Verify** |
|------------------------|-------------------------------|------------------------------|
| Taxable transaction: base amount × 5% = correct VAT, totals match | Exempt service / entity → 0% VAT, labelled “Exempt” | Inclusive/exclusive VAT treatment; rounding to 2 decimals |
| Fee calculation across credit card (2.5%), direct debit (1%), cheque (0%) – all correct | Deactivated service still appears for historical transactions with “Inactive” flag | Fee rule changes mid‑period apply correctly on boundaries |
| Tax invoice generation: all FTA‑mandatory fields present, TRN displayed, export to PDF | Duplicate receipt entry prevented or flagged | Tax invoice in Arabic: RTL layout, translated labels |
| Smart receipt fee: per‑receipt fee × quantity = total | Zero fee services not breaking totals | Service charges for loading: differentiate by instrument (card, account, voucher) |
| Aggregated fees report: sums of fee per fee type match detail | Orphan service rules (deleted) – handled gracefully | Cross‑report consistency: tax detail sum = tax summary total |

### Common Test Scenarios & Conditions

| Transaction Audit: full chronological log of status changes with user and timestamp | Non‑existent transaction ID → “Transaction not found” | Audit trail must include all state transitions; no missing steps |

| Smart receipt / e‑voucher report only shows transactions with those instruments | Report with zero amounts (adjustment) displayed without error | Data consistency: totals match source tables |

**Included Reports:**

- Shared revenues between DTPS and Sharjah Municipality
- Shared revenues between SEDD and SCTDA
- Shared revenues between Prevention & Safety Authority and SAND
- Shared fees summary between Sharjah Municipality and Service Centers
- Service List Report (government fees)
- POS Transactions by Revenue Source

### Common Test Scenarios & Conditions
| **Positive Scenarios** | **Negative / Edge Scenarios** | **Key Conditions to Verify** |
|------------------------|-------------------------------|------------------------------|
| Transaction assigned to revenue entity correctly based on service mapping | Changing revenue entity mid‑month – historical data unchanged, new transactions use new entity | Revenue share percentages applied correctly for each entity; total 100% |
| Shared revenue split: 50%–50% for DTPS/Municipality on designated services | No transactions for shared service → empty report or “No data” | Decimal precision on splits; rounding differences handled (e.g., 1 AED remainder) |
| Multiple entities: summary per entity with correct drill‑down | User with entity scope restriction sees only own entity data | Service fee list reflects master data: active/inactive, fees |
| Service center shared fees: 80% Municipality, 20% center | Center user sees only own center’s share | Revenue sharing rules effective date logic tested |

**Included Reports:**
- All **Tahseel Accounts Reports** (Card Statistic, Detailed Card, Transactions by Card, Aggregated Fees, SFD Detail, Lost Cards, Deduct & Load Requests, Voucher Receipt, Initial Balance, SFD Wallets, Card Transaction Details, Corporate Card Balance)

### Common Test Scenarios & Conditions
| **Positive Scenarios** | **Negative / Edge Scenarios** | **Key Conditions to Verify** |
|------------------------|-------------------------------|------------------------------|
| Card issuance, load, deduct, refund – full history in card transaction report | Insufficient balance for deduct/refund → rejected | Running balance calculation after each movement |
| E‑voucher batch loading: all vouchers activated, total value matches batch | Duplicate voucher load prevented | Load fee calculation: 1% of amount, minimum fee rule |
| Cheque clearing: status changes from Pending → Cleared reflected in reports | Cheque bounce: report shows “Returned” with reason | Masked account numbers; cheque amount = transaction amount |
| Lost card replacement: old card blocked, new card with transferred balance, report shows mapping | Non‑existent card/voucher search → “Not found” | Fee aggregation across instruments correct |
| Daily user loading report: per user, per centre, counts and totals accurate | Centre manager sees only own centre | Service charges report: zero charges if all exempt |

| Deposit summary per entity: total deposits, refunds, retains, net amount | No deposit accounts → “No active accounts” | Cross‑report: refund total in detailed report matches summary |

## Universal Test Conditions (Applicable to All Reports)

- **RBAC:** Verify that each user role sees only the reports permitted by their menu/scope; direct URL access blocked.
- **Localisation:** Arabic/English language switch updates all labels, data formatting (dates, numbers), and RTL layout in PDF.
- **Filters:** Boundary dates, future dates, special characters in text fields (SQL injection), invalid combinations.
- **Export:** PDF and Excel contain all columns, correct totals, appropriate logos/headers; no broken characters in Arabic.
- **Performance:** Report generation within acceptable time for typical and peak volumes; no timeout or memory errors.
- **Data Integrity:** Totals in summary reports match sum of detail reports; aggregated fees/taxes align with individual calculations.
- **Error Handling:** Graceful messages for “No data”, “Invalid input”, “Server error” without exposing stack traces.
- **Audit Trail:** Every report generation should be logged (user, timestamp, parameters) for compliance.
